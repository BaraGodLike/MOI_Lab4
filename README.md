# MOI3_5

Учебный рендерер на C++ с использованием **Embree 3** для трассировки лучей по треугольной сцене.  
Программа строит изображение 3D-сцены, считает локальное освещение и тени, а затем сохраняет результат в формате **PPM**.

## Назначение проекта

Проект реализует упрощённую схему рендеринга трехмерной сцены:

- сцена задаётся в коде;
- геометрия представляется треугольниками;
- пересечения лучей со сценой вычисляются с помощью Embree;
- цвет пикселя определяется локальной моделью освещения;
- изображение сохраняется в `PPM`.

Проект подходит как основа для описания:

- построения лучей камеры;
- проверки пересечений;
- вычисления теней;
- расчёта диффузной и зеркальной составляющих освещения;
- постобработки перед выводом изображения.

  - вызов функции сохранения в `PPM`.

  - структура сцены.

  Объявление функции создания сцены.

- `scene_config.cpp`  
  Главный файл настройки сцены:
  - положение камеры;
  - положение и размеры объектов;
  - количество объектов;
  - количество и параметры источников света;
  - цвета и материалы объектов.

  Объявление функции рендера.

- `renderer.cpp`  
  Основной алгоритм рендеринга:
  - создание геометрии;

  - запуск лучей;
  - вычисление пересечений и теней;
  - расчёт итогового цвета пикселя.

  Сохранение изображения в формат `PPM`, tone mapping и гамма-коррекция.

После сборки исполняемый файл будет расположен в:

```text
build/bin/Release/renderlab.exe
```

## Запуск программы

Пример запуска:

```powershell
./build/bin/Release/renderlab.exe --ppm result.ppm
```

Поддерживаемые аргументы:

- `--width N` — ширина изображения
- `--height N` — высота изображения
- `--ppm file.ppm` — имя выходного PPM-файла
- `--fov angle` — вертикальный угол обзора камеры

Пример:

```powershell
./build/bin/Release/renderlab.exe --width 1000 --height 800 --fov 35 --ppm scene.ppm
```

## Как задаётся сцена

Сцена задаётся в файле `scene_config.cpp`.

В этом файле можно менять:

- положение камеры;
- направление взгляда камеры;
- количество коробок;
- количество сфер;
- размеры коробок;
- радиусы сфер;
- количество источников света;
- цвета объектов;
- цвета света;
- интенсивности источников света.

### Камера

Камера задаётся так:

```cpp
scene.camera = {
    Pf(...),   // положение камеры
    Pf(...),   // точка, в которую камера смотрит
    Vf(...),   // направление "вверх"
};
```

### Коробки

Каждая коробка задаётся так:

```cpp
{
    Pf(...),            // origin
    Vf(...),            // size
    OMIT_...,           // какие грани убрать
    Matrix43f(...),     // transform
    {diffuse, specular, shininess},
}
```

### Сферы

Каждая сфера задаётся так:

```cpp
{
    Pf(...),            // center
    radius,
    depth,              // детализация сетки
    Matrix43f(...),
    {diffuse, specular, shininess},
}
```

### Источники света

Точечный источник:

```cpp
{LIGHT_POINT, color, position, Vf(0,0,0), intensity}
```

Направленный источник:

```cpp
{LIGHT_DIRECTIONAL, color, Pf(0,0,0), direction, intensity}
```

### Цвет и материал объекта

Материал задаётся так:

```cpp
{Vf(r, g, b), Vf(sr, sg, sb), shininess}
```

Где:

- первый `Vf(...)` — диффузный цвет;
- второй `Vf(...)` — цвет зеркальной составляющей;
- `shininess` — степень концентрации блика.

## Система координат

Все объекты задаются в **глобальных координатах сцены**.

В текущей конфигурации:

- `x` — глубина сцены, ближе/дальше относительно камеры;
- `y` — горизонтальная ось, влево/вправо;
- `z` — вертикальная ось, вверх/вниз.

Ось `z` используется как направление "вверх", так как камера задаётся с вектором:

```cpp
Vf(0.0f, 0.0f, 1.0f)
```

## Общая схема алгоритма

Алгоритм работы программы выглядит так:

1. В `scene_config.cpp` задаётся сцена.
2. Функция `MakeDefaultScene()` возвращает структуру `SceneDefinition`.
3. `main.cpp` передаёт сцену в `RenderScene(...)`.
4. В `renderer.cpp` объекты преобразуются в треугольную геометрию.
5. Для каждого пикселя строится первичный луч от камеры.
6. Embree вычисляет пересечение луча со сценой.
7. Если луч попал в поверхность, вычисляется локальное освещение.
8. Для каждого источника света строится теневой луч.
9. Если источник не перекрыт другим объектом, его вклад добавляется в цвет точки.
10. Итоговый массив RGB передаётся в `ppm.cpp`.
11. Выполняется нормировка яркости, гамма-коррекция и запись в `PPM`.

## Роль Embree в проекте

Embree используется как библиотека для ускоренного поиска пересечений лучей со сценой.

Embree выполняет:

- создание устройства трассировки;
- создание сцены;
- хранение треугольной геометрии;
- построение внутренней ускоряющей структуры;
- поиск пересечения первичных лучей;
- проверку теневых лучей на окклюзию.

Embree не выполняет:

- настройку сцены;
- выбор материалов;
- вычисление локального освещения;
- определение цвета пикселя;
- запись изображения в файл.

Эти действия выполняются кодом проекта.

## Как строятся лучи камеры

Для каждого пикселя `(row, col)` вычисляются координаты на плоскости изображения:

\[
s_x = \left(2 \cdot \frac{col + 0.5}{width} - 1\right)\cdot aspect \cdot \tan\left(\frac{fov_y}{2}\right)
\]

\[
s_y = \left(1 - 2 \cdot \frac{row + 0.5}{height}\right)\cdot \tan\left(\frac{fov_y}{2}\right)
\]

Затем вычисляется направление луча:

\[
\vec d = \operatorname{normalize}(\vec forward + s_x \vec right + s_y \vec up)
\]

где:

- `forward` — направление взгляда камеры;
- `right` — горизонтальный базис камеры;
- `up` — вертикальный базис камеры.

Луч задаётся в параметрической форме:

\[
\vec r(t) = \vec o + t\vec d
\]

где:

- \(\vec o\) — положение камеры;
- \(\vec d\) — направление луча.

## Как считаются тени

После того как найдено пересечение луча камеры с объектом, для каждого источника света строится **теневой луч**.

Начало теневого луча слегка смещается вдоль нормали:

\[
\vec p_{shadow} = \vec p + \epsilon \vec n
\]

где:

- \(\vec p\) — точка пересечения;
- \(\vec n\) — нормаль;
- \(\epsilon\) — малое смещение (`kShadowBias`).

Это смещение нужно, чтобы луч не пересекал ту же поверхность повторно из-за численных ошибок.

### Для точечного источника

Направление на свет:

\[
\vec l = \frac{\vec p_{light} - \vec p}{\|\vec p_{light} - \vec p\|}
\]

Расстояние до света:

\[
d = \|\vec p_{light} - \vec p\|
\]

### Для направленного источника

Используется фиксированное направление:

\[
\vec l = \operatorname{normalize}(-\vec direction)
\]

После этого вызывается проверка окклюзии через Embree.  
Если между точкой и источником света есть другой объект, свет в эту точку не попадает.

## Модель освещения

Используется **локальная модель освещения**:

- диффузная составляющая;
- зеркальная составляющая;
- жёсткие тени.

### Диффузная составляющая

Диффузный вклад рассчитывается по формуле Ламберта:

\[
L_d = (\vec k_d \odot \vec c_{light}) \cdot A \cdot \max(0, \vec n \cdot \vec l)
\]

где:

- \(\vec k_d\) — диффузный цвет материала;
- \(\vec c_{light}\) — цвет источника света;
- \(A\) — коэффициент затухания;
- \(\vec n\) — нормаль;
- \(\vec l\) — направление на источник света.

### Зеркальная составляющая

Используется half-vector:

\[
\vec h = \operatorname{normalize}(\vec l + \vec v)
\]

\[
spec = \max(0, \vec n \cdot \vec h)^{shininess}
\]

\[
L_s = (\vec k_s \odot \vec c_{light}) \cdot A \cdot spec
\]

где:

- \(\vec v\) — направление на камеру;
- \(\vec k_s\) — цвет зеркальной составляющей.

### Затухание точечного света

Для точечного источника используется обратноквадратичное затухание:

\[
A = \frac{I}{d^2}
\]

где:

- \(I\) — интенсивность источника;
- \(d\) — расстояние до точки.

## Как строится геометрия

### Коробка

Коробка не передаётся в Embree как специальный примитив.  
Сначала она вручную разбивается на треугольники:

- задаются 8 вершин;
- затем формируются индексы треугольников;
- каждая грань коробки представляется двумя треугольниками.

То есть разбиение коробки на треугольники выполняет код проекта.

### Сфера

Сфера также строится вручную:

1. создаётся базовая многогранная аппроксимация;
2. каждый треугольник рекурсивно делится;
3. новые вершины нормализуются;
4. после этого вершины масштабируются до нужного радиуса.

Итоговая сфера тоже передаётся в Embree как треугольная сетка.

## Как формируется итоговый PPM

После рендеринга получается массив RGB-значений в вещественном формате.

Перед записью в `PPM` выполняются:

1. анализ диапазона яркостей;
2. нормировка по гистограмме;
3. гамма-коррекция;
4. перевод в диапазон `0..255`.

### Вычисление яркости пикселя

Для каждого пикселя берётся:

\[
rgb_{max} = \max(R, G, B)
\]

### Нормировка яркости

Строится гистограмма по 1024 корзинам и выбирается масштаб так, чтобы типичный средний уровень яркости оказался около `0.5`.

Используемый масштаб:

\[
scale = \frac{0.5}{dh \cdot (i128 + 0.5)}
\]

### Гамма-коррекция

После масштабирования применяется:

\[
V_{out} = 255 \cdot \min(1, V^{1/2.2})
\]

Формат записи — текстовый `PPM P3`.

## Ограничения текущей реализации

- используется локальная модель освещения;
- отсутствуют вторичные отражённые лучи;
- отсутствует глобальное освещение;
- отсутствуют мягкие тени от площадных источников;
- сцена задаётся только кодом, без загрузки внешних моделей;
- поддерживаются только коробки и сферы.
